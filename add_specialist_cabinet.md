# План реализации: Кабинет Специалиста (Specialist Dashboard)

Функционал **полностью оправдан и не является избыточным**. Это ядро работы врача/специалиста: ведение базы пациентов, отслеживание динамики (записи по датам) и ручная корректировка ИИ-рекомендаций, как и требует главное правило приложения (ИИ — только помощник).

## Сравнение подходов: Кастомная разработка vs. Headless CMS (на примере Hygraph)

Прежде чем приступать к разработке собственной базы пациентов и интерфейса, стоит рассмотреть альтернативу — использование готовых Headless CMS (например, Hygraph, Strapi или Directus).

### Вариант А: Использование Hygraph (или другой CMS)
**Плюсы:**
- **Быстрый старт бэкенда:** Не нужно писать SQL-миграции, создавать таблицы `clients` и писать CRUD API (`GET /api/clients` и т.д.). База и GraphQL API генерируются автоматически.
- **Готовая админка:** Вы сразу получаете готовый интерфейс для просмотра списков клиентов, редактирования полей и фильтрации. Не нужно верстать таблицы и формы.
- **Управление контентом:** Удобно хранить библиотеку упражнений, редактировать тексты, добавлять видео-инструкции без участия программиста.

**Минусы:**
- **Сложность интеграции с камерой и ИИ:** Hygraph — это хранилище данных. Флоу съемки с гироскопом, взаимодействие с Gemini и Telegram-ботом все равно придется писать кастомно и связывать через webhooks с CMS.
- **Отчуждаемость данных:** Hygraph — это облачный SaaS. В ТЗ (раздел 4) у нас заявлено "Независимый сервер — полный контроль над данными" и "Возможность локальной установки для клиник". С Hygraph локальная установка невозможна (либо очень дорога). Медицинские данные (фото в трусах, диагнозы) будут храниться на сторонних серверах в Европе/США, что может нарушать законы о персональных данных.
- **Ограничения интерфейса:** Админка CMS универсальна. Сделать в ней специфичный "таймлайн осанки", встроить кастомный холст с отрисовкой углов перекоса таза поверх фото — крайне сложно или невозможно.

### Вариант Б: Кастомная разработка (Текущий план)
**Плюсы:**
- **Полный контроль над данными:** База PostgreSQL стоит на вашем сервере. Вы можете развернуть всё локально в клинике. Это идеальное соответствие требованиям безопасности.
- **Бесшовный интерфейс:** Кабинет специалиста выглядит как единое современное приложение с камерой. Можно реализовать любые визуальные фишки (наложение фото До/После с прозрачностью, спец-редактор рекомендаций).
- **Специфичная бизнес-логика:** Легко реализовать такие функции, как "Новые заявки из Telegram бота", привязку Telegram ID к клиенту клиники и автоматическую отправку PDF-отчета обратно в чат.

**Минусы:**
- **Больше времени на разработку:** Придется вручную верстать интерфейс кабинета, писать SQL-запросы и API-методы для управления базой пациентов.
- **Сложнее управлять общим контентом:** Если вы захотите добавить новое упражнение в справочник, это придется делать через БД или писать для этого отдельный интерфейс.

**Вывод для нашего проекта:** 
Учитывая, что приложение работает с интимными фотографиями клиентов и специфичным процессом съемки/ИИ-анализа, **кастомный подход (Вариант Б) является более безопасным и гибким**. Использование Hygraph может ускорить создание базы, но создаст проблемы с приватностью медицинских данных и ограничит нас в визуализации результатов анализа.

## User Review Required
> [!IMPORTANT]
> **Жизненный цикл онлайн-заявок (Согласовано):**
> 1. Клиент делает фото дома. Если у него **базовая подписка**, он просто получает ответ ИИ.
> 2. Если подписка **премиальная**, заявка попадает в общий пул "Новые премиум-заявки".
> 3. Этот пул видят **все зарегистрированные специалисты** в своем кабинете. Специалист может "взять" заявку, и тогда этот клиент привязывается к нему (попадает в его личную базу).
> 
> На начальном этапе мы сымитируем этот процесс: добавим в БД поле статуса/типа подписки, чтобы протестировать появление премиум-заявок в общем пуле.

### Бэкенд API (Backend)

#### [MODIFY] `server.py`
- `GET /specialist`: Маршрут для отдачи файла `specialist.html`.
- `POST /api/clients`: Создание нового оффлайн-клиента врачом.
- `GET /api/clients`: Поиск "своих" клиентов.
- `GET /api/pool`: Получение списка свободных премиум-заявок (где `analysis_type = 'premium'` и нет привязки к специалисту).
- `POST /api/analyses/<id>/assign`: Взять анализ из пула себе в работу.
- `GET /api/clients/<id>`: Получение карточки клиента и истории его анализов.
- `PUT /api/analyses/<id>/recommendations`: Сохранение изменений в списке.

### Фронтенд: UI и Логика (Frontend)
Кабинет специалиста будет вынесен в полностью независимое приложение (отдельный HTML и JS), чтобы не перегружать камеру для обычных пользователей.

#### [MODIFY] `index.html` и `app.js` (Форма клиента)
- На экране анкеты (`#form-screen`) добавить новый элемент интерфейса — переключатель (или карточки) для выбора типа анализа:
  - **Базовый (Бесплатно)**: Только отчет от Искусственного Интеллекта.
  - **Премиум (С проверкой врача)**: Детальный разбор снимков и утверждение плана реабилитологом.
- Передача выбранного `analysis_type` в `/form/save_draft`.

#### [NEW] `specialist.html`
- Отдельный HTML-файл для интерфейса врача. Будет использовать те же CSS стили, но иметь собственную структуру:

#### [NEW] `specialist_dashboard.js`

## Verification Plan

### Automated Tests
- Запуск Flask-сервера и проверка отсутствия синтаксических ошибок в `server.py` и `database.py`.

### Manual Verification
- Авторизоваться под ролью администратора/специалиста.
- Убедиться, что открывается Кабинет (Dashboard).
- Создать нового клиента (например, "Иван Иванов").
- Найти его через поиск (ввести "Иван").
- Открыть карточку, нажать "Провести анализ", пройти флоу съемки, дождаться ответа ИИ.
- Убедиться, что анализ появился в карточке клиента.
- Открыть результаты анализа, отредактировать текст рекомендаций ИИ и нажать "Сохранить".
- Перезагрузить страницу и проверить, что изменения рекомендаций сохранились.
